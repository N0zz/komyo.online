# i18n — string handling

The komyo i18n system is **live** (`i18n.js` + `gamekit.t`). Generated games are **i18n-native**: every
player-visible string goes through `KIT.t(key, { def: 'English' })`. Full design: [`plans/i18n-plan.md`](/Users/kkolodziejczyk/arcade/plans/i18n-plan.md).

## The rule

**No raw English literals in player-facing UI — wrap them in `KIT.t`.**

```js
KIT.t('game.<slug>.menuTitle', { def: 'BRICK BREAKER' })      // game-specific
KIT.t('game.common.playAgain', { def: 'Play Again' })          // shared across games
KIT.t('game.common.scoreLine', { score: score, def: 'Score {score}' })  // interpolated
```

- **`def:` is the English source.** English renders from `def:` even when no key exists in `i18n.js`,
  so a freshly generated game is fully playable in English with **zero** `i18n.js` edits. Translations
  are added later (see below). Never rely on a key already existing — always pass `def:`.
- **Keys:** game-specific strings use `game.<slug>.*`; strings shared across games use `game.common.*`
  (already translated — reuse them, don't re-invent). Common ones available: `menu`, `paused`, `resume`,
  `quitToMenu`, `best`, `score`, `time`, `gameOver`, `move`, `pause`, `drag`, `scoreLine` (`{score}`),
  `play`, `playAgain`, `modeLabel`, `click`, `tap`, `esc`.
- **Interpolation:** put values in the params object and reference them as `{name}` in the string —
  `KIT.t('game.<slug>.reachedWave', { wave: w, def: 'Reached wave {wave}' })`. Never build sentences by
  `+`-joining fragments (word order changes between languages).
- **Plurals:** pass `{ count: n }`; the value can be a plural object `{ one, few, many, other }` in
  `i18n.js`. `def:` can't express plurals, so if a string pluralizes, add at least an `en`+`pl` entry.
- **Keep emoji/glyphs outside the translatable slot** where practical (`'🏆 ' + KIT.t(...)`).
- **`.toUpperCase()` is fine on a `t()` result** for a title, but the cleaner path is an already-upper
  `def:`/translation (matches the shipped games).

## Storage labels stay stable — don't translate them

The best-store / `record:` **`mode`** label is a storage key (`gamekit_pb`), not display text. Keep it a
stable English string (`const MODE_LABEL = 'Classic'`). Translate the *displayed* mode name separately
(a `modeDisplay()` that `t()`s it), like `games/breakout/`. Same for challenge/cosmetic ids — the id is
never translated; only its rendered label.

## Wiring (the template already does this)

- `i18n.js` is in the atomic `<head>` (after `cosmetics.js`) **and** in the `sw.js` `SHELL` array, in
  lockstep — a missing one silently kills translations offline.
- `game.<slug>.title` is the single source for the display name; `share:`/`shareText` and (later)
  challenges/cosmetics reference that key rather than re-stating the name.

## Translations — Polish is REQUIRED (a test enforces it)

`node test.mjs` has an **i18n coverage** check: the `pl` locale must contain **every** referenced key
(literal `t()`/`data-t` keys + each game's `game.<slug>.title`/`.blurb` + every `cos.*` name/desc). So a
new game/string that lacks a Polish entry **fails the suite** — that's how "shipped English-only by
accident" is caught. After generating a game you MUST add its `game.<slug>.*` keys to the `pl` block in
`i18n.js` (reuse `game.common.*` for shared strings — those are already translated). The failing test
lists exactly which keys are missing.

- **`en`** needs no entry (that's what `def:` is for) — EXCEPT plural keys (`{one,few,many,other}`), which
  must exist in `en` since `def:` can't pluralize.
- **`es/pt/fr/it`** are "soon" locales: each must be **empty OR complete** (a full superset of `pl`). Leave
  them empty until you're doing a full translation pass — a half-filled locale fails the test.
- Scaffolding does NOT auto-edit `i18n.js` (mechanical string surgery on a shared file is too fragile) —
  add the `pl` keys by hand as the final step.
