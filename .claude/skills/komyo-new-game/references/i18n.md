# i18n — string handling (stub)

The komyo i18n system is **not shipped yet**. This file documents what the skill does today and how
it flips when i18n lands. Full design: [`/Users/kkolodziejczyk/arcade/plans/i18n-plan.md`](/Users/kkolodziejczyk/arcade/plans/i18n-plan.md).

## TODAY — emit English-inline, but keyable-ready

Generated games write **English strings inline** in `index.html` (no `KIT.t`, no `i18n.js` yet).
Write them so a later extraction pass is mechanical:

- **Prefer whole display strings over concatenation.** Write the full sentence as one literal, not
  pieces glued at runtime — `'You cleared the board!'`, not `'You cleared ' + n + ' rows'` split
  across fragments.
- **Keep interpolated values separable.** When a value must appear mid-string, keep it as a single
  clean insertion point (`'Reached wave ' + w`) — trivially convertible to a parameterized template
  (`t('game.<slug>.reachedWave', {wave: w})` → `'Reached wave {wave}'`). Avoid building sentences
  from many `+` joins or embedding numbers inside larger phrases in ways that won't survive
  reordering in another language.
- Keep emoji/glyphs outside the translatable text where practical.

## FUTURE — when the i18n system ships

Once `i18n.js` + `gamekit.t` exist, the skill switches to i18n-native generation:

- Templates emit `KIT.t('game.<slug>.*', {params})` keyed strings instead of English literals.
- Add `i18n.js` to the atomic `<head>` (after `cosmetics.js`) **and** to the `sw.js` `SHELL` array,
  in lockstep — a missing one silently kills translations offline.
- The scaffold appends the new game's `en` keys (and stub/MT `pl/es/pt/fr`) to `i18n.js`.
- `game.<slug>.title`/`.blurb` become the single source for the display name; challenges/cosmetics
  reference the key rather than re-stating the name.

## Cross-dependency — keep the two plans in sync

This skill and the i18n plan overlap on per-game strings. **Whichever ships FIRST, the SECOND is
updated to match** (see the coordination section in each plan):

- **i18n ships first:** the skill is authored i18n-native from the start (the "English-inline"
  decision above is dropped); templates emit `t()` and the scaffold writes keys.
- **This skill ships first:** it generates English-inline games; the i18n rollout then extracts
  those games' strings into `i18n.js` as part of its per-game extraction tasks.

Whoever lands second must be revised so the generated games and the i18n catalogue stay consistent.
