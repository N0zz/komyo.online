# i18n — string handling

The komyo i18n system is **live** and **split per language**: `i18n.js` is the loader + the `en`
(def-source) dict; every other locale lives in its own root file `i18n.<code>.js` (the `pl` reference
in `i18n.pl.js`); `gamekit.t` reads the merged `window.KOMYO_I18N`. Generated games are **i18n-native**:
every player-visible string goes through `KIT.t(key, { def: 'English' })`. Full design:
[`plans/i18n-plan.md`](/Users/kkolodziejczyk/arcade/plans/i18n-plan.md).

## The rule

**No raw English literals in player-facing UI — wrap them in `KIT.t`.**

```js
KIT.t('game.<slug>.menuTitle', { def: 'BRICK BREAKER' })      // game-specific
KIT.t('game.common.playAgain', { def: 'Play Again' })          // shared across games
KIT.t('game.common.scoreLine', { score: score, def: 'Score {score}' })  // interpolated
```

- **`def:` is the English source — en defs live inline in code.** English renders from `def:` even
  when no key exists in the i18n catalogue, so a freshly generated game is fully playable in English
  with **zero** i18n-file edits. Translations are added later (see below). Never rely on a key
  already existing — always pass `def:`.
- **Keys:** game-specific strings use `game.<slug>.*`; strings shared across games use `game.common.*`
  (already translated — reuse them, don't re-invent). Common ones available: `menu`, `paused`, `resume`,
  `quitToMenu`, `best`, `score`, `time`, `gameOver`, `move`, `pause`, `drag`, `scoreLine` (`{score}`),
  `play`, `playAgain`, `modeLabel`, `click`, `tap`, `esc`.
- **Interpolation:** put values in the params object and reference them as `{name}` in the string —
  `KIT.t('game.<slug>.reachedWave', { wave: w, def: 'Reached wave {wave}' })`. Never build sentences by
  `+`-joining fragments (word order changes between languages).
- **Plurals:** pass `{ count: n }`; the value can be a plural object `{ one, few, many, other }` in
  the locale files. `def:` can't express plurals, so if a string pluralizes, add at least an `en`
  entry (in `i18n.js`'s `en` dict) + a `pl` entry (in `i18n.pl.js`).
- **Keep emoji/glyphs outside the translatable slot** where practical (`'🏆 ' + KIT.t(...)`).
- **`.toUpperCase()` is fine on a `t()` result** for a title, but the cleaner path is an already-upper
  `def:`/translation (matches the shipped games).

## Storage labels stay stable — don't translate them

The best-store / `record:` **`mode`** label is a storage key (`gamekit_pb`), not display text. Keep it a
stable English string (`const MODE_LABEL = 'Classic'`). Translate the *displayed* mode name separately
(a `modeDisplay()` that `t()`s it), like `games/breakout/`. Same for challenge/cosmetic ids — the id is
never translated; only its rendered label.

## Wiring (the template already does this)

- `i18n.js` (the loader) is in the atomic `<head>` (after `cosmetics.js`) — the tag is unchanged by
  the per-language split; the loader pulls in the active locale's `i18n.<code>.js` itself. The ROOT
  `sw.js` `SHELL` (the one site-wide service worker — no per-game sw.js) must list `i18n.js` **AND
  every `i18n.<code>.js` locale file**, in lockstep — a missing one silently kills translations (or
  a whole language) offline (test-enforced).
- `game.<slug>.title` is the single source for the display name; `share:`/`shareText` and (later)
  challenges/cosmetics reference that key rather than re-stating the name.

## Translations — every populated locale is enforced (a test checks it)

`node test.mjs` has an **i18n coverage** check: the `pl` locale must contain **every** referenced key
(literal `t()`/`data-t` keys + each game's `game.<slug>.title`/`.blurb` + every `cos.*` name/desc), and
every OTHER non-`en` locale must be **empty or a complete superset of `pl`**. So a new game/string that
lacks a Polish entry **fails the suite** — that's how "shipped English-only by accident" is caught —
and once `pl` has the new keys, every other *populated* locale needs them too or the suite goes red on
that locale. After generating a game you MUST:

1. Add its `game.<slug>.*` keys to **`i18n.pl.js`** — the `pl` reference locale's own file (reuse
   `game.common.*` for shared strings — those are already translated). En defs stay inline in code
   (`def:`). The failing test lists exactly which keys are missing.
2. Hand off to the **komyo-i18n-translate** skill (incremental mode: "add the new keys to all existing
   complete locales") so every other populated locale's `i18n.<code>.js` receives them.

- **Never hardcode the locale list.** Discover the currently-configured locales at runtime by merging
  `i18n.js` + every `i18n.<code>.js` (or read `window.KOMYO_I18N_AVAILABLE` from `i18n.js` + `'en'`,
  or `gamekit.langs()`), and treat every populated non-`en` locale as a required target:

  ```bash
  cd ~/arcade && node -e "const fs=require('node:fs'),vm=require('node:vm');const sb={window:{}};sb.globalThis=sb;for(const f of ['i18n.js'].concat(fs.readdirSync('.').filter(f=>/^i18n\.[a-z]{2}\.js$/.test(f)).sort()))vm.runInNewContext(fs.readFileSync(f,'utf8'),sb,{filename:f});for(const[k,v]of Object.entries(sb.window.KOMYO_I18N))console.log(k,Object.keys(v).length)"
  ```

  A future language must never require an edit to this skill.
- **`en`** needs no entry (that's what `def:` is for) — EXCEPT plural keys (`{one,few,many,other}`), which
  must exist in `en` (the dict in `i18n.js`) since `def:` can't pluralize.
- A locale whose `i18n.<code>.js` map is still empty may legitimately stay empty — "empty OR complete
  superset" is the enforced invariant; a half-filled locale fails the test.
- Scaffolding does NOT auto-edit the i18n files (mechanical string surgery on shared files is too
  fragile) — add the `pl` keys to `i18n.pl.js` by hand, then run the translate skill for the rest.
